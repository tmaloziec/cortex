## Installing

Clone the repository, then run the launcher â€” it creates the venv
and installs pinned dependencies on first run.

```bash
git clone https://github.com/build-on-ai/cortex.git
cd cortex
./run.sh agent
```

First-run output:

```
Creating venv...
[GPU] Model: gemma4:26b (auto)
  + Policy Engine loaded (8 tools)
  + Recovery Engine (fallback: none)
  + Context Compactor (limit: 32000 tokens)
```

Requires Python 3.12+, a reachable Ollama instance at
`http://localhost:11434` (or `OLLAMA_URL`), and â€” if you want the
web UI â€” a browser on the same machine.

## Running modes

`./run.sh` supports three modes.

### `agent` â€” interactive CLI (default)

```bash
./run.sh agent
```

Chat in the terminal. Tool calls (`bash`, `read_file`, `write_file`,
`grep_search`, `glob_find`, `list_dir`) run against the real machine
gated by the Policy Engine. DENY rules kill the call; ASK rules
prompt you to confirm with `t/n`; ALLOW runs without prompt.

Useful in-session commands (type at the prompt):

| Command      | Effect                                   |
|:-------------|:-----------------------------------------|
| `/help`      | List all slash commands                  |
| `/policy`    | Show active Policy Engine rules          |
| `/tokens`    | Show token usage of current context      |
| `/briefing`  | Fetch fresh briefing from CS (if configured) |
| `/status`    | Connectivity + model + CS status         |
| `/clear`     | Reset conversation history, keep system prompt |
| `/think`     | Toggle thinking mode (if model supports) |
| `exit`       | Quit and save session                    |

Sessions auto-save to `~/.cortex/sessions/<timestamp>.json`; use
`/load <timestamp>` to restore, or the `/api/sessions` endpoint via
the web UI.

Plugin mode:

```bash
./run.sh agent --mode <plugin_name>
```

Switches the agent into a plugin's system prompt with its tool
surface activated. See `PLUGIN_GUIDE.md`.

### `web` â€” browser UI

```bash
./run.sh web
```

Starts FastAPI + WebSocket on `http://localhost:8080`. The launcher
prints a bootstrap URL:

```
Open this URL in your browser:
  http://127.0.0.1:8080/?token=<auto-generated-32-byte-token>
```

The first `GET /?token=â€¦` exchanges the URL token for an HttpOnly
session cookie and redirects to `/`. After that, the URL in the
address bar is clean and the cookie handles authentication for
WS + `/api/*` calls. The UI stores nothing in JavaScript state â€”
auth lives entirely in the cookie.

**Environment variables you'll actually use:**

| Variable | Default | Meaning |
|:--|:--|:--|
| `WEB_HOST` | `127.0.0.1` | Bind address. Do NOT set to `0.0.0.0` unless you also set `WEB_INSECURE=1` or `WEB_TOKEN`. |
| `WEB_PORT` | `8080` | Port. |
| `WEB_TOKEN` | auto-generated | Override the 32-byte token. Empty string requires `WEB_INSECURE=1`. |
| `WEB_INSECURE` | unset | Set to `1` to allow `WEB_TOKEN=""` (auth disabled). Use only on dev localhost. |
| `MAX_WS_CONNECTIONS` | 10 | Concurrent WS session cap. Clamped to `[1, 1000]`. |
| `CORTEX_TRUST_PROXY_HEADERS` | unset | Set to `1` when running behind nginx / Cloudflare / caddy so `True-Client-IP` / `CF-Connecting-IP` / `X-Real-IP` is honoured for rate-limit keying. Do NOT set when uvicorn is network-reachable directly. |
| `WEB_ORIGIN_ALLOWLIST` | empty | Comma-separated extra Origins allowed for WS CSRF check. |
| `OLLAMA_URL` | `http://localhost:11434` | Ollama endpoint. |
| `OLLAMA_MODEL` | auto (GPU / CPU detect) | Model name. |
| `ANTHROPIC_API_KEY` | unset | Anthropic fallback (disabled unless `CORTEX_FALLBACK_ANTHROPIC=1` is also set). |
| `CORTEX_FALLBACK_ANTHROPIC` | `0` | Enable silent upload to Anthropic on Ollama connection error. Every upload is WARN-logged with payload size. |
| `CORTEX_FALLBACK_REDACT_TOOL_OUTPUTS` | `0` | When fallback is enabled, strip untrusted containers (tool output, briefing, compacted history) before the upload leaves the host. |

WebSocket endpoint: `ws://localhost:8080/ws` (add `?token=â€¦` for
CLI clients; browsers get cookie automatically).

REST endpoints (all require `Depends(_require_auth_dep)` except
`/`, `/health`, `/api/logout`, and `/ws`):

| Method | Path | Purpose |
|:--|:--|:--|
| `GET` | `/` | Bootstrap HTML; exchanges token for cookie. |
| `GET` | `/health` | Liveness probe, `{"status":"ok"}`. |
| `POST` | `/api/logout` | Revoke current session + clear cookie. |
| `GET` | `/api/models` | List Ollama models + RAM-fit hint. |
| `POST` | `/api/model` | Switch active model. Body: `{"model":"name"}`. |
| `GET` | `/api/sessions` | List saved sessions. |
| `GET` | `/api/session/{id}` | Load one session. |
| `DELETE` | `/api/session/{id}` | Delete session. |

### `worker` â€” autonomous CS polling

```bash
./run.sh worker
./run.sh worker --once   # one task then exit
```

The worker registers in the Consciousness Server, polls
`/api/tasks/pending/<AGENT_NAME>` every `POLL_INTERVAL` seconds
(default 10s), executes whatever it finds, reports results back via
`/api/tasks/<id>/status` and `/api/notes`. No human in the loop; the
Policy Engine's ASK path is downgraded to SKIP so the worker never
blocks on confirmation. Set `CS_URL` (see next section) for this mode
to have anything to do.

**Environment variables specific to worker mode:**

| Variable | Default | Meaning |
|:--|:--|:--|
| `CS_URL` | empty | Consciousness Server URL, e.g. `http://your-cs-host:3032`. |
| `AGENT_NAME` | `cortex` | Registered name in CS. |
| `AGENT_LOCATION` | `local` | Free-form label (e.g. `adax`, `laptop`). |
| `POLL_INTERVAL` | `10` | Seconds between CS polls. |

## Consciousness Server (CS) integration

CS is a separate service (your infrastructure's agent registry +
task queue + memory store) at an `http(s)://â€¦` URL. Cortex talks
to it over plain HTTP, always as the client â€” CS does not call
Cortex.

Set it up once:

```bash
export CS_URL="http://your-cs-host:3032"    # your CS endpoint
export AGENT_NAME="cortex-laptop"            # unique per agent
```

Then any mode that has CS features turned on will use it. With
CS configured:

- **agent / web**: `/briefing` command fetches agent-specific
  context on session start. `cs_note`, `cs_task`, `cs_briefing`
  tools become available.
- **worker**: registers on start, heartbeats every loop, polls
  for tasks.

Without CS (CS_URL unset), Cortex is a standalone local agent â€”
nothing breaks, CS features are silently disabled.

`validate_cs_url` at startup rejects malformed `CS_URL` or URLs
with embedded `user:pass@` (credentials in URLs leak to access
logs).

**Transport is plaintext HTTP today.** See "Security notes" below
before deploying across networks.

## Policy Engine

The Policy Engine (`policy.py`) gates every tool call. Default rules
live in `DEFAULT_POLICIES` and cover:

- **DENY**: `rm -rf /`, `mkfs`, fork bombs, pipe-to-shell, shell
  reverse connections, writes to `~/.ssh/`, `~/.aws/`, shell init
  files, cron, plugin directories.
- **ALLOW**: everything not denied (Claude-Code-style; no
  per-command ASK by default).

Override with a JSON file and point to it:

```bash
export POLICY_FILE=~/.cortex/policy.json
```

Example overlay:

```json
{
  "bash": {
    "deny": [
      "docker\\s+run\\s+--privileged"
    ]
  }
}
```

Your rules are prepended to the defaults, so you can only *add*
restrictions. Shared lists (`_CREDENTIAL_DENY`, `_PERSISTENCE_DENY`,
`_HISTORY_DENY`, `_SYSTEM_DIRS_DENY`) are re-expanded across every
tool that inherits from them â€” you don't have to list
`read_file` / `grep_search` / `glob_find` / `list_dir` separately.

Type errors in a custom policy (e.g. `"deny": "..."` instead of a
list) are logged and the bad tool is skipped â€” the rest of your
overlay still loads.

## Sessions

Sessions persist to `~/.cortex/sessions/<timestamp>.json`. The
message list in each file is plain JSON â€” you can grep it, load
it into another tool, or back it up. Beware: **untrusted content
lives in the `content` fields** (tool output inside `<tool_output_
<nonce> untrusted="true">` containers); don't `cat` a session file
into another LLM without understanding that the wrapping is what
keeps it data instead of instructions.

Sessions are also mirrored to CS (`/api/memory/conversations`) if
`CS_URL` is set.

Delete cleanly: `DELETE /api/session/<id>` via REST, or manually
from the disk.

## Troubleshooting

**"Ollama unreachable" on startup.**
Start Ollama: `systemctl start ollama` (Linux) or `ollama serve`.
Verify: `curl -sf http://localhost:11434/api/tags`.

**Bootstrap URL gives 401.**
The token in the URL doesn't match `WEB_TOKEN`. The auto-generated
token lives only in the launcher's stdout â€” you can't recover a lost
one without restarting.

**"Session rate limit exceeded" (429) during normal use.**
600 requests per session per 60 seconds. If a real workload needs
more, file a bug with the pattern â€” the limit targets scripted
cookie replay, not humans.

**Worker never picks up tasks.**
Check `CS_URL` reachable, `AGENT_NAME` matches what CS expects, and
the pending-task endpoint returns non-empty JSON when queried
manually. Logs are at `./worker.log` by default.

**Plugin doesn't appear in `/help`.**
`PLUGIN_NAME` must match `^[A-Za-z0-9_.-]{1,64}$` and filename must
match `^[A-Za-z][A-Za-z0-9_]*\.py$`. Check startup log:

```
discover_plugins: skipping plugin with non-conforming name: ...
```

Paths with `..` are refused.

**Want to clear session + try again clean.**
CLI: `/clear`. Web: reload with `POST /api/logout` first.

## Security notes for users

- Treat the bootstrap URL as secret. It's shorter-lived than a
  master token (single use before becoming a cookie), but the full
  URL still appears in shell history / tmux scrollback / asciinema
  recordings.
- Do not set `WEB_HOST=0.0.0.0` without understanding what you've
  just exposed. Cortex has shell access to its host.
- Plugins are trusted code. Read the Python before dropping it in
  `plugins/`.
- `CORTEX_FALLBACK_ANTHROPIC=1` uploads every message (including
  tool output, file contents, CS briefing) to `api.anthropic.com`
  on every Ollama ConnectionError. Enable only if you've accepted
  that trade-off. Add `CORTEX_FALLBACK_REDACT_TOOL_OUTPUTS=1` if
  you want the tool-output bodies stripped first.
- `CS_URL=http://â€¦` sends briefings, notes, tasks, and session
  transcripts over plaintext HTTP. See `ARCHITECTURE.md` â†’
  "CS transport" for the current state and TLS plans.
